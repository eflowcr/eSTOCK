# W6B — Notifications UI

## Scope
- FN1: Bell icon in topbar with unread badge + dropdown (last 10, polling 60s)
- FN2: /notifications page with filters + pagination
- FN3: Preferences matrix in /settings (notifications tab)

## Decisions

### Dropdown implementation
Custom click-outside pattern (same as user menu in topbar) — no CDK Overlay.
Rationale: CDK not used anywhere in codebase; the existing pattern is simple and consistent.

### Relative time
Custom `formatRelativeTime()` util in the bell component — no external library (date-fns, dayjs).
Rationale: simple bucketed logic (mins/hours/days/weeks) covers all notification timescales.

### /settings notifications tab integration
FN3 adds the preferences matrix to the *existing* `/settings` route `notifications` tab.
The existing tab already has global user-prefs toggles (email/push/marketing from UserPreferences). 
The new matrix (per event_type × in_app/email/push) is added BELOW as a separate card.

### Push column
Disabled (all checkboxes disabled with opacity-50 + cursor-not-allowed + tooltip "S5").
Model already supports `push: boolean`; UI reflects future-ready state.

### W5 services base
notification.model.ts and notifications.service.ts ported from s2/w5-models-services branch.
The service's startPolling/stopPolling helpers are retained but the bell component manages
its own setInterval per spec (component-level polling, not service-level).

### upsertPreferences uses PUT
Backend endpoint is `PUT /api/notifications/preferences`; uses FetchService.put().
W5 service used `.post()` — corrected to `.put()`.
